Project Overview

Design Philosophy

01

Zero Tick Dependency

The entire AI runs on UE5's Behaviour Tree framework with services and decorators handling perception. The SCP-173 Character has PrimaryActorTick disabled — no Actor Tick overhead at all. All logic executes through the BT's own update cycle.

02

NavMesh Pathfinding

Movement uses AAIController::MoveToActor and MoveToLocation rather than manual SetActorLocation. This gives proper slope traversal, stair climbing, and obstacle avoidance through Unreal's built-in navigation system.

03

Reverse Perception

Unlike standard AI perception (where AI detects targets), SCP-173 needs the reverse: can the player's camera see SCP-173? A custom system was built instead of using UAIPerceptionComponent because the built-in component traces from AI eyes outward — the wrong direction entirely.

04

Teleport via SetActorLocation

The teleport mechanic deliberately uses SetActorLocation with ETeleportType::TeleportPhysics instead of NavMesh movement. This ensures SCP-173 can teleport to any valid location regardless of NavMesh coverage — a critical failsafe that must work everywhere.

05

Designer-Friendly Configurables

Every gameplay-critical value is exposed as an EditAnywhere UPROPERTY with appropriate clamp ranges and unit labels. Designers can tune speeds, distances, timings, and probabilities directly in the Blueprint editor without touching C++.

06

Modular BT Nodes

11 custom C++ classes (3 services, 2 decorators, 6 tasks) each handle a single responsibility. The movement tasks share a common abstract base class. Nodes are reusable and independently testable.

Technical Highlights

Engineering Decisions

Key architectural choices that separate this from a basic gameplay implementation. Each decision was made with production scalability, performance, and designer workflow in mind.

Zero Actor Tick

SCP-173's character has PrimaryActorTick disabled. All AI logic runs through the Behaviour Tree's own update cycle — services tick on configurable intervals, decorators evaluate per-frame only when needed. No wasted CPU on idle frames.

Reverse Perception

Observation is checked from the player's camera to the AI, not the other way around. This required a custom system instead of UE5's UAIPerceptionComponent, which only traces outward from the AI's perspective.

Behaviour Tree Priority Arbitration

Five branches with FlowAbortMode::Both create an implicit priority system. Kill aborts everything, Freeze aborts movement, Teleport aborts navigation — no manual state machine transitions needed.

NavMesh-Aware Movement

All pathfinding uses MoveToActor / MoveToLocation through UE5's navigation system. Slopes, stairs, and obstacles are handled automatically. Teleport deliberately bypasses NavMesh using SetActorLocation as a failsafe.

Event-Driven Audio & VFX

Sound and particles are triggered by state transitions inside SetCurrentState(), not by polling. Movement sound starts/stops on state change, dust VFX spawns at ground-traced locations with a configurable delay.

Modular Node Architecture

11 custom BT classes with single-responsibility design. Movement tasks inherit from a shared abstract base, overriding only StartNavMovement(). Focus management, path monitoring, and abort handling are centralized and reused.

System Architecture

Behaviour Tree Structure

The entire AI runs through a single Behaviour Tree with 5 priority-ordered branches. Higher-priority branches automatically abort lower ones. The system is built from 11 custom C++ classes — 3 services, 2 decorators, and 6 tasks.

System Flow

Priority Branches

Branch 1 Highest

Kill Player

Triggers instantly when within attack distance.

Branch 2 High

Freeze In Place

Locks SCP-173 when observed. Stops movement, clears focus, spawns dust VFX after a configurable delay.

Branch 3 Medium

Teleport Behind Player

Failsafe repositioning when unobserved too long. Uses SetActorLocation (works without NavMesh).

Branch 4 Low

Flank Reposition

Probability-based lateral movement to create unpredictable encounters.

Branch 5 Fallback

Move Toward Player

Default direct approach via NavMesh pathfinding.

Services & Shared Infrastructure

Perception Service

Camera FOV + LOS checks. Updates all Blackboard keys: observation state, distance, unobserved timer, attack range, and teleport readiness.

Reaction Window Service

Counts down the post-teleport timer. Triggers kill if the player doesn't look in time.

Movement Base Class

Shared abstract base for both movement tasks. Handles focus management, path status monitoring, and movement sound lifecycle. Subclasses only override the move request.

Core Behaviour

Freeze When Observed, Move When Unseen

The fundamental SCP-173 mechanic: complete stillness under observation, rapid movement the instant line of sight breaks. The transition is instantaneous — there is no blend or interpolation delay.

Clip 1 — Core Freeze/Move Loop

Demonstrates the fundamental observation cycle. Looking directly at SCP-173 keeps it frozen in place. Turning away allows it to rapidly close the gap using NavMesh pathfinding. Snapping back reveals it has repositioned — now frozen again in its new position. The movement speed of 2000 cm/s gives the impression of sudden repositioning rather than gradual walking.

How It Works

1

BTDecorator_IsObserved runs a per-frame check using a dot-product FOV test combined with a line trace from the player's camera to SCP-173's location.

2

If observed, the decorator activates Branch 2 (Freeze) which aborts any active movement tasks via FlowAbortMode::Both.

3

BTTask_FreezeInPlace immediately stops NavMesh pathfinding, clears the AI focus, and after a configurable FXDelay triggers the Observed state (spawning dust VFX).

4

When the player looks away, the decorator deactivates, and Branch 5 (Move) resumes — calling MoveToActor(Player) with NavMesh pathfinding.

Line-of-Sight Detection

Camera-Based Observation System

Observation is determined from the player's camera, not the AI. This is the reverse of standard AI perception — the question is "can the player see SCP-173?" not "can the AI see the player." This is why we built a custom system rather than using Unreal's UAIPerceptionComponent.

Clip 2 — LOS Obstruction

Shows SCP-173 moving when a wall or obstacle blocks the player's line of sight. Even though the player is facing SCP-173's general direction, the raycast hits the obstacle first, confirming SCP-173 as unobserved. Sidestepping to regain clear sight freezes it again instantly.

Detection Logic

The system uses a two-step validation in SCP173FunctionLibrary::IsActorObserved:

1

FOV Check: Calculate the dot product between the camera's forward vector and the direction to SCP-173. If the angle exceeds PlayerFOVHalfAngle, SCP-173 is outside the player's field of view and considered unobserved.

2

LOS Raycast: Fire a LineTraceSingleByChannel from the camera location to SCP-173's location. If the trace hits any obstacle before reaching SCP-173, it is considered unobserved (occluded). Only a direct, unobstructed hit confirms observation.

Why Not UAIPerceptionComponent?

  • Wrong direction — Built-in perception traces from the AI's eyes outward. SCP-173 needs the reverse: trace from the player's camera to the AI.
  • Latency — The perception component fires stimulus events asynchronously with its own tick interval. SCP-173 requires per-frame accuracy — a single frame of delay means it moves while being watched.
  • Overhead — The perception system supports multiple senses (sight, hearing, damage, touch), team affiliation, and stimulus aging. SCP-173 needs exactly one check. Custom is lighter and more precise.

Teleport Mechanic

Failsafe Repositioning

Teleportation is not part of SCP-173's normal movement. It is a failsafe that triggers only when SCP-173 has been unobserved for an extended duration or cannot find a valid path. This ensures the player can never permanently evade SCP-173 by exploiting geometry.

Clip 3 — Teleport + Survival

The player looks away for the teleport timeout duration. SCP-173 teleports behind the player (audio cue + camera shake alert). The player quickly turns around and spots SCP-173 within the reaction window, freezing it and surviving.

Teleport timeout set to 4s for demonstration.

Clip 4 — Teleport Kill

Same teleport trigger, but this time the player fails to turn around within the reaction window. The timer expires and SCP-173 executes the kill sequence. The audio cue and camera shake serve as the player's only warning.

Teleport timeout set to 4s for demonstration.

Teleport Location Selection

SCP173FunctionLibrary::FindTeleportLocation uses a multi-candidate approach:

1

Generate candidate positions behind and around the player at TeleportDistanceBehindPlayer distance.

2

Validate each candidate is not within the player's FOV using IsLocationInFOV.

3

Run a LOS trace from the player to each candidate — prefer positions with no direct line of sight.

4

Execute teleport via SetActorLocation with ETeleportType::TeleportPhysics — works regardless of NavMesh coverage.

Player Kill Condition

Two Paths to Death

SCP-173 can kill through two independent mechanisms: proximity-based direct attack and post-teleport reaction timeout. Both converge on the same TriggerKill() function which broadcasts the kill event and notifies Blueprint.

Clip 5 — Direct Attack Kill

SCP-173 approaches the player through normal unobserved movement. Once it enters the AttackDistance radius, the highest-priority BT branch fires immediately, triggering the kill sequence. At 2000 cm/s movement speed, this happens rapidly once the player loses line of sight at close range.

Path A: Direct Attack

BTService_SCP173Perception continuously measures distance. When distance falls below AttackDistance, it sets bInAttackRange = true on the Blackboard. BTDecorator_AttackRange activates Branch 1 (highest priority), which runs BTTask_KillPlayer.

Path B: Teleport Reaction Timeout

After teleporting, BTService_ReactionWindow decrements ReactionTimer each tick. If the timer reaches zero before the player looks at SCP-173, the service calls TriggerKill(). If the player does look in time, BTTask_FreezeInPlace cancels the reaction window by resetting the timer and clearing bReactionActive.

Reposition Behaviour

Unpredictable Flanking

Instead of always moving directly toward the player, SCP-173 occasionally repositions to flanking angles. This prevents players from easily predicting its position and creates more tense encounters where it can appear from unexpected directions.

Clip 6 — Flanking Behaviour

Repeatedly looking away and back shows SCP-173 arriving from different angles rather than always dead-ahead. The flanking chance is set to 50% for this demonstration, with a max flank angle of 60 degrees, creating clearly visible lateral repositioning.

FlankChance set to 0.5 for demonstration.

How Flanking Works

1

BTService_FlankDecision rolls a random value each service tick. If it falls below FlankChance, flanking is activated.

2

The service computes a flank target by rotating the player-to-SCP173 direction vector by a random angle between -MaxFlankAngle and +MaxFlankAngle degrees.

3

The computed position is written to the FlankTarget Blackboard key and bShouldFlank is set to true.

4

BTTask_MoveToFlankPosition calls MoveToLocation(FlankTarget) using NavMesh pathfinding. If the flank target is invalid, it gracefully falls back to MoveToActor(Player).

Visual Feedback

Dust Displacement Effect

When SCP-173 becomes visible after moving, a dust particle effect spawns at its feet to suggest it was in motion just before the player looked. The effect uses a downward line trace to ensure particles appear on the actual ground surface regardless of terrain.

Clip 7 — Dust VFX

The player turns away and quickly snaps back. A dust poof appears at SCP-173's feet as it freezes, implying it was caught mid-motion. The VFX spawns at the ground surface detected by a downward line trace, not at the actor's center.

Ground Detection

GetGroundLocation() performs a downward LineTraceSingleByChannel (500cm) from the actor's location on ECC_Visibility. The impact point is used as the VFX spawn location, ensuring particles appear on the actual floor surface — whether flat ground, slopes, or stairs. Falls back to the bottom of the capsule if the trace misses.

FX Delay

The dust VFX does not spawn instantly when observation begins. BTTask_FreezeInPlace waits for the configurable FXDelay (read from the pawn) before triggering the Observed state transition. This creates a subtle "just caught it" moment. If the decorator aborts the task during the delay (player looks away again), the VFX never fires — preventing false triggers.

Camera Effects

Proximity Shake & Teleport Alert

Camera effects serve as non-visual threat indicators. The proximity shake builds tension as SCP-173 closes in, while the teleport shake provides an urgent alert when it appears behind the player.

Clip 8 — Camera Shake Proximity

Walking toward a frozen SCP-173 shows the camera shake intensifying as the distance closes. Moving backward shows the shake fading. The effect uses a multi-axis sine wave offset on the SpringArm component with distance-based intensity scaling.

Proximity Shake

Driven by BTService_SCP173Perception updating DistanceToPlayer on the Blackboard. The player character reads this distance and computes a ShakeIntensity that scales from zero (at CameraShakeStartDistance) to MaxCameraShakeIntensity (at distance zero). The shake is applied as a multi-axis sine wave offset: X uses CameraShakeFrequency, Y uses a 1.3x multiplier, Z uses 0.7x — creating an organic, non-repeating tremor.

Teleport Shake

A one-shot tension burst triggered by TriggerTeleportCameraShake() when SCP-173 teleports. Uses TeleportShakeIntensity and TeleportShakeDuration for a quick, sharp jolt that alerts the player to look behind them.

Audio Feedback

Three Layers of Sound

Audio serves as a critical gameplay feedback channel. Each sound layer communicates a different aspect of SCP-173's state, giving the player spatial awareness without needing to look directly at the threat.

Clip 9 — Audio Showcase

A continuous take demonstrating all three audio layers: the ambient presence audio fading in as the player approaches SCP-173, the looping stone scraping sound when SCP-173 moves while unobserved, and the distinct teleport audio cue when the failsafe triggers.

Ambient Presence

A persistent UAudioComponent (AmbientAudioComponent) attached to SCP-173. Volume scales inversely with distance using UpdateAmbientAudio() — louder when closer, silent beyond AmbientAudioMaxDistance. Auto-plays on BeginPlay if a sound asset is assigned.

Movement Sound (Looping)

A second UAudioComponent (MovementAudioComponent) that plays a looping stone scraping sound. Automatically starts when SCP-173 transitions to the Unobserved state and stops when it transitions to Observed. Controlled via StartMovementSound() / StopMovementSound() inside SetCurrentState().

Teleport Cue

A one-shot PlaySoundAtLocation fired when SCP-173 teleports. This is the player's primary alert that SCP-173 is now behind them. Played via PlayTeleportSound() in BTTask_TeleportBehindPlayer.

Robustness

Edge Cases Handled

Production AI must handle every corner case gracefully. Each scenario below was identified during development and has a dedicated solution built into the system.

Teleport FOV Validation

Before teleporting, the system double-checks that the destination is outside the player's camera FOV using IsLocationInFOV(). If the teleport location would be visible, the teleport is rejected and SCP-173 continues normal movement instead of appearing in plain sight.

Geometry Collision Prevention

Teleport uses FindTeleportLocation() which sweeps behind the player and validates with line traces to ensure no geometry blocks the destination. Returns MAX_FLT sentinel on failure, causing a graceful fallback to standard movement.

Post-Teleport Reaction Window

After teleporting behind the player, a configurable reaction timer begins. If the player turns around in time, the timer cancels and SCP-173 freezes normally. If not, the reaction service triggers a kill. This prevents unfair instant deaths while maintaining threat tension.

Instant Movement Interruption

All movement task decorators use FlowAbortMode::Both. The moment the player's camera detects SCP-173, the BT immediately aborts the active movement task, calls StopMovement() and ClearFocus(), and activates the freeze branch — no frame delay.

Ground-Traced VFX Placement

Dust VFX doesn't spawn at the capsule center. A 500cm downward LineTraceSingleByChannel finds the actual ground surface. Fallback: capsule half-height subtraction if the trace misses. Works correctly on slopes, stairs, and uneven terrain.

Focus Lifecycle Management

AI face-tracking (SetFocus) activates during movement and post-teleport, and is properly cleared (ClearFocus) on freeze and movement completion. Prevents SCP-173 from rotating toward the player while frozen, which would break the illusion of a statue.

AI State Logic

State Machine via Behaviour Tree

SCP-173's states are not driven by a traditional state machine. Instead, the Behaviour Tree's priority-based branch selection naturally produces the correct state transitions through decorator conditions and Blackboard keys.

Observed

SCP-173 is frozen and cannot move. Triggered when the player's camera has direct, unobstructed line of sight within the FOV half-angle. Movement sound stops, NavMesh pathfinding is halted, AI focus is cleared. After the configurable FX delay, dust VFX spawns at the ground and BP_OnObserved fires.

Trigger: BTDecorator_IsObserved = true

Unobserved

SCP-173 actively moves toward or repositions around the player using NavMesh pathfinding. Movement sound loops, AI focus is set to the player (face-tracking), and the unobserved timer begins counting toward the teleport threshold.

Trigger: BTDecorator_IsObserved = false

Teleport Ready

Line of sight has been broken for longer than TeleportTimeout. SCP-173 teleports behind the player via SetActorLocation, activates the reaction window, plays the teleport audio cue, and triggers camera shake on the player.

Trigger: UnobservedTime ≥ TeleportTimeout

Attack Range

SCP-173 is within AttackDistance of the player. The highest-priority BT branch fires immediately, executing the kill sequence via TriggerKill(). This overrides all other states.

Trigger: DistanceToPlayer ≤ AttackDistance
